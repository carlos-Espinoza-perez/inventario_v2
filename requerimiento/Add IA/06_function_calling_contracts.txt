===========================================================================
🎯 OBJETIVO DE ESTE MÓDULO
===========================================================================
Definir los "Contratos de Función". Estos son los esquemas JSON estrictos 
que Gemma 2B debe emitir cuando identifique una intención de ACCIÓN o 
CONSULTA. Actúan como el puente entre el lenguaje natural y tus Repositorios.

===========================================================================
🛠️ CATÁLOGO DE FUNCIONES DISPONIBLES
===========================================================================

Gemma solo tiene permiso para "invocar" estas 3 estructuras:

1. registrar_venta:
   {
     "function": "registrar_venta",
     "args": {
       "producto_busqueda": "String",
       "cantidad": "int",
       "cliente_nombre": "String?",
       "tipo_pago": "contado | credito"
     }
   }

2. consultar_existencia:
   {
     "function": "consultar_existencia",
     "args": {
       "producto_busqueda": "String",
       "bodega_nombre": "String?"
     }
   }

3. registrar_entrada:
   {
     "function": "registrar_entrada",
     "args": {
       "producto_busqueda": "String",
       "cantidad": "int",
       "costo_unidad": "double?"
     }
   }

===========================================================================
🏗️ PROMPT DE INYECCIÓN DE HERRAMIENTAS (SYSTEM PROMPT)
===========================================================================

Para que Gemma sepa usar estas herramientas, inyectamos este texto:

"Tienes acceso a las siguientes herramientas para interactuar con el inventario:
- registrar_venta: Úsala cuando el usuario quiera vender o despachar.
- consultar_existencia: Úsala para preguntas sobre stock o precios.
- registrar_entrada: Úsala cuando llegue mercadería nueva.

Si la información es insuficiente (ej. 'vende galletas' sin decir cuántas), 
tu respuesta debe pedir el dato faltante en lugar de generar el JSON."

===========================================================================
💻 VALIDACIÓN EN DART (MODELO DE DATOS)
===========================================================================

class AICallContext {
  final String functionName;
  final Map<String, dynamic> arguments;

  AICallContext({required this.functionName, required this.arguments});

  factory AICallContext.fromJson(Map<String, dynamic> json) {
    return AICallContext(
      functionName: json['function'],
      arguments: json['args'] ?? {},
    );
  }
}

===========================================================================
⚠️ ANÁLISIS AUTOCRÍTICO (REVISIÓN SENIOR)
===========================================================================

* AMBIGÜEDAD DE NOMBRES: El usuario dirá "Coca" pero en Drift se llama 
  "Coca Cola 500ml". Por eso el campo se llama 'producto_busqueda'. 
  Ese string se enviará al Motor de Embeddings (Paso 04) para buscar 
  el ID real en Drift. Nunca confíes en que la IA de el ID exacto.

* TIPADO DE DATOS: Gemma a veces envía números como Strings ("5" en vez de 5). 
  El parser de Dart debe ser robusto: 'int.tryParse(args["cantidad"].toString())'.

* SEGURIDAD: Ninguna función de este contrato debe borrar datos. 
  El borrado (Delete) siempre debe ser manual por seguridad.

* ARQUITECTURA: Estas funciones no llaman directamente al Repositorio, sino 
  a la capa de **Use Cases** (`RegistrarVentaUseCase`, etc.). Esto garantiza 
  que la IA ejecute las mismas validaciones y lógica que la interfaz manual.
